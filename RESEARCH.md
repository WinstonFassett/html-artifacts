# Single-File HTML Artifacts — Research Findings

> Round 1: 106-agent workflow, 23 sources, 111 claims, 25 verified (13 confirmed / 12 killed). Import maps, CDNs, browser support.
> Round 2: 100-agent workflow, 18 sources, 83 claims, 25 verified (16 confirmed / 9 killed). Agentic artifact ecosystem, platforms, skills, prior art.
> Date: 2026-06-12

---

## Summary

Single-file HTML artifacts that run with no build step are a well-established pattern. The core enabling technology is **ES module import maps**, baseline-stable across all major browsers since March 2023. The practical modern stack:

1. Inline `<script type="importmap">` block (hand-written or generated by JSPM)
2. `<script type="module">` using bare specifiers resolved to CDN URLs
3. Optionally, `esm.sh/tsx` (~1.7KB) for inline JSX/TypeScript

## Confirmed Findings

### 1. Import Maps — Browser Support
Import maps are baseline-stable across all major browsers (since March 2023). No polyfill needed for modern browsers; `es-module-shims` covers the ~4–6% on older browsers. Multiple-import-map support (Chrome 133+, Safari 18.4+) is still behind flags in Firefox as of mid-2026.

**Sources:** [MDN](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Elements/script/type/importmap) · [SpiderMonkey blog](https://spidermonkey.dev/blog/2023/02/23/javascript-import-maps-part-1-introduction.html) · [Lit/Shoelace buildless guide](https://dev.to/matsuuu/buildless-workflow-through-import-maps-featuring-lit-shoelace-and-more-4ill)

### 2. Import Maps — Hard Constraints
Three rules practitioners must know:
1. The `<script type="importmap">` block **must appear before** any `<script type="module">` that uses its specifiers. Violating this throws: `"An import map is added after module script load was triggered."`
2. Import maps **only apply to `import` statements and `import()`** — not `<script src>`, workers, or worklets.
3. **Prefix remapping requires a trailing slash** on the map key (e.g., `"lit/": "https://..."` not `"lit"`).

**Sources:** [MDN](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Elements/script/type/importmap) · [SpiderMonkey blog](https://spidermonkey.dev/blog/2023/02/23/javascript-import-maps-part-1-introduction.html)

### 3. JSPM Generator — Practical Tooling
JSPM provides two ways to generate import maps:
- **Visual generator** at [generator.jspm.io](https://generator.jspm.io/) — has an "HTML Boilerplate" checkbox (default: checked) that outputs a ready-to-run single HTML file.
- **Hosted API** at `https://api.jspm.io/generate` — GET or POST, returns `{ staticDeps, dynamicDeps, map }` JSON.

**Sources:** [generator.jspm.io](https://generator.jspm.io/) · [jspm.org/cdn/api](https://jspm.org/cdn/api)

### 4. esm.sh — Two Key Primitives
- **`esm.sh/tsx`** (~1.7KB): enables inline JSX/TypeScript via `<script type="text/babel|text/jsx|text/ts|text/tsx">` with zero build tooling.
- **`?external` parameter**: marks peer dependencies for browser-side import-map resolution rather than bundling them in. Example: `'react': 'https://esm.sh/react@19.2.4'` with `?external=react` and `*react-dom`.

> Note: The basic URL import pattern (`import React from 'https://esm.sh/react@18'`) was refuted 0–3 by adversarial verifiers. Use the import map + `?external` pattern instead and verify against current [esm.sh docs](https://esm.sh/).

**Source:** [github.com/esm-dev/esm.sh](https://github.com/esm-dev/esm.sh)

### 5. esm.run (jsDelivr ESM CDN) — Operational
`https://esm.run/lit` → 301 → `cdn.jsdelivr.net/npm/lit/+esm` serving Lit 3.3.3 ESM (verified June 2026). Pattern: `"lit": "https://esm.run/lit"` in import map.

> Caveat: esm.run explicitly states it is **not intended for production use** — convenience alias only, no SLA.

**Source:** [Lit/Shoelace buildless guide](https://dev.to/matsuuu/buildless-workflow-through-import-maps-featuring-lit-shoelace-and-more-4ill)

### 6. Vanilla No-Dep Pattern (juliensimon/browser-games)
The canonical zero-dependency game pattern: HTML5 Canvas + Web Audio API, no frameworks, no CDN imports. `juliensimon/browser-games` covers pong, asteroids, pac-man, space-invaders. Note: these use a 3-file structure (index.html + style.css + game.js), not true single-file. Pattern is still valid as a starting point for inlining.

**Source:** [github.com/juliensimon/browser-games](https://github.com/juliensimon/browser-games)

## Import Map Boilerplate

```html
<!DOCTYPE html>
<html>
<head>
  <script type="importmap">
  {
    "imports": {
      "lit": "https://esm.run/lit",
      "lit/": "https://esm.run/lit/"
    }
  }
  </script>
</head>
<body>
  <script type="module">
    import { html, LitElement } from 'lit';
    // ...
  </script>
</body>
</html>
```

For JSX/TypeScript (esm.sh/tsx):
```html
<script src="https://esm.sh/tsx"></script>
<script type="text/tsx">
  import { useState } from 'https://esm.sh/react@19'
  // JSX here
</script>
```

## Known Gaps / Open Questions

1. **Hosting platform constraints** (Claude artifacts, Observable, CodePen, Val Town, StackBlitz) — size limits, CSP headers, allowed CDNs — were **not confirmed** by any surviving claim. Need direct investigation.
2. **`file://` URL support** for import maps — does it work in Chrome/Firefox/Safari without a local server? Research was inconclusive (1-2 split, not confirmed).
3. **Production CDN recommendation** — esm.run discourages production use; need clarity on esm.sh vs jspm.io vs jsdelivr for production single-file artifacts.
4. **Genre/category taxonomy** — the claim that single-file HTML repos span 7+ genres was refuted 0-3. No primary-source taxonomy exists; genre lists are anecdotal.
5. **Curated GitHub awesome-lists** specifically for single-file HTML artifacts — not found. `github.com/topics/single-html-file` and `github.com/topics/single-file?l=html` exist but weren't confirmed as comprehensive collections.
6. **Vanilla HTML platforms** — are there tools specifically targeting vanilla HTML generation (not React/JSX) that have emerged in 2026?
7. **CDN library version corpus** — no systematic analysis of what Mermaid/Chart.js/Tailwind/Alpine CDN patterns AI tools actually emit in practice.

## How vibes.diy runs React with no build (source-verified)

Inspected directly from the cloned `platforms/vibes.diy` source. The runtime is **Sucrase + import map + esm.sh fallback** — NOT esbuild-wasm or Parcel:

1. AI writes `App.jsx` (TSX, `export default function App()`)
2. **Sucrase** (`pkg/slack/transform-sucrase.ts`) strips TS types + transforms JSX → `React.createElement` *in the browser at runtime*. Sucrase is ~20KB, fast because it only strips/transforms, never type-checks.
3. **`bare-specifier-rewrite.ts`** rewrites any unrecognized bare import (`three`, `chart.js`, `tone`) → `https://esm.sh/<name>`. Only rewrites the top-of-file import region; specifiers already in the import map (or matched by trailing-slash prefix) are left untouched.
4. The transformed code runs as a blob module; `mount-vibes.ts` reuses a persistent React root so a broken intermediate render during streaming doesn't blank the iframe.

Key files: `vibes.diy/vibe/runtime/bare-specifier-rewrite.ts`, `pkg/slack/transform-sucrase.ts`, `vibe/runtime/mount-vibes.ts`.

## CommonJS / file://-safe JSX (answer to "non-ESM way to do TSX")

Two ways to run JSX with **no ESM, works from `file://`**:

| Approach | Mechanism | Size | Notes |
|----------|-----------|------|-------|
| **Babel Standalone** | UMD globals + `<script type="text/babel" data-presets="react,typescript">` | ~2.7MB | Full JSX/TSX. React/ReactDOM from `window`. Slower (JS parse). |
| **htm** | Tagged template literals, `html\`<App/>\`` | ~600B | No transform at all. JSX-*like*, not real JSX. |

vs. **esm.sh/tsx** (ESM): ~1.7KB loader, real TSX, needs HTTP (uses `import.meta.url` + fetch). MUST be loaded as `<script type="module">` and needs `react/jsx-runtime` in the import map.

## Round 5 — Reactive view stacks + power libs (build-tested, not just researched)

The key insight: **any *runtime* React library works no-build via esm.sh + esm.sh/tsx.** Compiler-dependent tools are the only exceptions.

### View / reactive stacks

| Stack | Reactivity | Loading | file://? | Verdict |
|-------|-----------|---------|----------|---------|
| React (esm.sh/tsx) | hooks, real TSX | ESM | ✗ | ✅ tested |
| React (Babel Standalone) | hooks, real JSX | script | ✓ | ✅ tested |
| React (htm) | hooks, tagged templates | script | ✓ | ✅ tested |
| Preact + htm | hooks, 3KB | ESM | ✗ | ✅ tested (needs import map for externalized `preact`) |
| Solid + `solid-js/html` | fine-grained signals | ESM | ✗ | ✅ tested |
| Vue 3 (full global) | composition API | script | ✓ | ✅ tested |
| petite-vue | directives, 6KB | ESM | ✗ | ✅ tested |
| Alpine | directives | script | ✓ | ✅ tested |
| Lit | reactive web components | ESM | ✗ | ✅ tested (shadow DOM) |
| **Svelte** | — | — | — | ❌ **compiler-required** |

### Compiler-required gaps (the things that DON'T work no-build)

- **Svelte** — confirmed: `compile()` converts `.svelte` → JS module; raw `.svelte` can't run in-browser. `svelte-browser-import` bundles the compiler but uses `eval`, is dev-only, Svelte 4 only. Svelte 5 explicitly can't be CDN-bundled (maintainer: "Resolving all the imports necessary is a huge pain"). Source: sveltejs/svelte#15658, svelte.dev/docs/svelte-compiler.
- **Solid via TSX** — esm.sh/tsx uses swc's generic jsx-runtime; Solid needs compile-time `babel-preset-solid` for fine-grained reactivity. The TSX path would silently produce non-reactive Solid. **Use `solid-js/html` tagged templates instead.**
- **Remotion rendering** — Player previews in-browser fine (tested: composition plays with timeline/controls), but mp4 export needs Remotion's Node render toolchain.

### Power / capability libs (build-tested)

| Capability | Library | Loading | file://? | Verdict |
|-----------|---------|---------|----------|---------|
| Declarative 3D in React | React Three Fiber v9 | ESM via esm.sh/tsx | ✗ | ✅ tested — `?external=react,react-dom,three` |
| Programmatic video preview | Remotion Player v4 | ESM via esm.sh/tsx | ✗ | ✅ tested — needs `remotion/no-react` in import map |
| In-browser SQLite | SQL.js 1.14 | script + WASM | ✓ | ✅ tested — `locateFile` → CDN for the .wasm |
| Local-first persistence | Fireproof | ESM | ✗ | ✅ tested (persists to IndexedDB) |

### esm.sh externalization gotcha

When a package is loaded with `?external=react` (etc.), esm.sh keeps the bare specifier (`react`, `preact`, `remotion/no-react`) instead of inlining it — so you **must** provide an import map entry for each externalized specifier, including deep sub-paths the library imports internally (e.g. `remotion/no-react`, `preact/hooks`). Missing one → `Failed to resolve module specifier`.

## Naming convention for this archive

- **`-esm` suffix** = uses ES module imports → needs an HTTP server (breaks from raw `file://` due to CORS on module loads)
- **no suffix** = plain `<script>` tags / UMD globals → runs directly from `file://`

The gallery (`index.html`) shows this as a badge per card: orange `ESM·http` / green `file://`.
