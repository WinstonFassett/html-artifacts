Firefly Backend Handlers v1 — fetch · scheduled · onChange

One file, three trigger patterns — server-side logic for every vibe

What this is. /backend.js is a server-side file that lives alongside /App.jsx and /access.js in a vibe's filesystem. Each named export maps to a trigger type: fetch handles incoming HTTP requests, scheduled runs on a timer, and onChange reacts to database writes. A config export declares scheduling intervals. All three handlers receive the same ctx object with database access, secrets, and user identity. The server discovers /backend.js automatically at push time — no deployment step, no infrastructure configuration.

Relationship to access.js. The access function (/access.js) is the gatekeeper — it runs synchronously on every write, validates permissions, and routes documents to channels. Backend handlers are reactors — they run asynchronously after writes land, on external HTTP triggers, or on a schedule. Access functions block writes; backend handlers respond to events. They share the same vibe filesystem and the same ctx.db interface for reading and writing data.

One file, familiar shape. The export names match Cloudflare Workers (fetch + scheduled) plus onChange for database reactivity. Most backends need a mix — a payment vibe needs webhook receipt (fetch) plus email-on-purchase (onChange). A single file with named exports is simpler to generate, simpler to reason about, and matches the mental model of "my app has a frontend file and a backend file."

Identity passthrough. Every ctx.db.put() writes as a specific userHandle, so the access function always sees a real user — no special system-write knobs needed. The default identity comes from the trigger: onChange passes through the original writer's handle, fetch uses the session-authenticated user, and scheduled defaults to the vibe owner. A handler can override with { as: userHandle }, but only the vibe owner's code can impersonate — the runtime enforces this. This keeps access.js clean: it validates the same user.userHandle field regardless of whether the write came from the frontend or a backend handler.

Secrets hierarchy. Secrets resolve with vibe-level overriding account-level overriding platform-level. A vibe sets WEBHOOK_URL at the vibe level; the user's SENDGRID_API_KEY comes from their account; a shared WEATHER_API_KEY comes from the platform. When a vibe is remixed, it runs with the remixer's secret hierarchy — the original creator's keys are never exposed.

Handler 1 — fetch(request, ctx)

External HTTP → process → respond — webhooks, OAuth, REST APIs

The fetch handler receives inbound HTTP requests at the vibe's published URL with a _api reserved path. The platform recognizes _api in the vibe's URL and routes to the BackendDO instead of serving static files. The handler gets the standard Request object as-is and returns a standard Response. URL parsing, path routing, signature verification, endpoint structure — all application concerns handled inside the handler. ctx.userInfo is null for webhook requests (no session); the handler writes as the vibe owner by default.

⚡ /backend.js — fetch handler (OAuth + webhook example)
export async function fetch(request, ctx) { const url = new URL(request.url) // OAuth callback — exchange code for token, store in DB if (url.pathname === "/oauth/callback") { const code = url.searchParams.get("code") const token = await exchangeToken(code, ctx.secrets.CLIENT_SECRET) await ctx.db.put({ _id: "oauth-token", ...token }) return new Response("Connected!") } // Stripe webhook — verify signature, process payment if (url.pathname === "/webhook/stripe") { const sig = request.headers.get("stripe-signature") const payload = await request.text() const event = verifyStripeSignature(payload, sig, ctx.secrets.STRIPE_WEBHOOK_SECRET) await ctx.db.put({ _id: `payment-${event.id}`, type: "payment", ...event.data }) return new Response("OK") } return new Response("Not Found", { status: 404 }) }

URL routing. Each vibe's fetch handler is reachable at /vibe/{ownerHandle}/{appSlug}/api/*. The Worker routes requests matching this pattern to the vibe's BackendDO. The path seen by the handler is relative to /api/ — a request to /vibe/alice/sonos-display/api/oauth/callback arrives as url.pathname === "/oauth/callback". Query parameters, headers, and body are passed through unchanged.

Handler 2 — scheduled(event, ctx)

Timer fires → poll external API → write to DB → frontend syncs

The scheduled handler runs on a repeating interval declared in the config export. Under the hood, this uses Durable Object alarms — each vibe's BackendDO sets an alarm for the next tick. The handler receives an event with the scheduled time and can read/write the database and make outbound HTTP requests. ctx.userInfo is null in scheduled handlers — there is no user context for timer-driven work.

⚡ /backend.js — scheduled handler (Sonos polling example)
export const config = { scheduled: { interval: "5s" } } export async function scheduled(event, ctx) { // read stored OAuth token const tokenDoc = await ctx.db.get("oauth-token") if (!tokenDoc) return // not connected yet // poll Sonos API const res = await fetch("https://api.ws.sonos.com/control/api/v1/playback", { headers: { Authorization: `Bearer ${tokenDoc.accessToken}` } }) const data = await res.json() // write current state — frontend picks up via live sync await ctx.db.put({ _id: "now-playing", track: data.track.name, artist: data.track.artist, albumArt: data.track.imageUrl, isPlaying: data.playbackState === "PLAYING", updatedAt: Date.now() }) }

config — schedule declaration
config.scheduled
interval: "5s" | "30s" | "1m" | "5m" | "15m" | "1h"  // minimum 5s, maximum 1h

Interval limits. The minimum interval is 5 seconds — faster polling is rejected at push time. The maximum is 1 hour. The interval is a hint, not a guarantee: if the handler takes longer than the interval to complete, the next tick starts immediately after the previous one finishes. There is no concurrent execution — only one invocation of scheduled runs at a time per vibe.

Handler 3 — onChange(event, ctx)

Database write lands → side effect fires — notifications, external sync, logging

The onChange handler runs asynchronously after a document write succeeds (including deletes). It receives the new document, the previous version (if any), the database name, and the identity of the writer. The handler runs after the access function has approved the write and the document is stored — it cannot block or reject the write. This is the right place for side effects that should not gate the write path: sending emails, posting to Slack, syncing to an external CRM.

⚡ /backend.js — onChange handler (order confirmation email)
export async function onChange(event, ctx) { // event.doc — current document (null on hard delete) // event.oldDoc — previous version (null on create) // event.dbName — which database // event.userInfo — who made the change (null for system writes) const isCreate = event.doc && !event.oldDoc if (event.doc?.type === "order" && isCreate) { await fetch("https://api.sendgrid.com/v3/mail/send", { method: "POST", headers: { Authorization: `Bearer ${ctx.secrets.SENDGRID_API_KEY}`, "Content-Type": "application/json" }, body: JSON.stringify({ to: event.doc.customerEmail, subject: "Order Confirmation", text: `Your order #${event.doc._id} has been placed.` }) }) } }

Use Case — Sonos Now-Playing Display

OAuth + polling + live sync — all three handlers in one vibe

A wall-mounted display showing what's currently playing on Sonos speakers. The backend handles the OAuth flow to connect the user's Sonos account, polls the playback API every 5 seconds, and writes the current track to the database. The frontend subscribes to live changes and renders the album art and track info. No onChange needed — but a more complex version could use it to log play history or sync to a scrobbling service.

Use Case — E-Commerce with Stripe

Webhook receipt + order confirmation email — fetch + onChange

An e-commerce vibe that receives Stripe webhook events and sends confirmation emails. The fetch handler verifies the Stripe signature and writes payment records to the database. The onChange handler watches for new payment documents and sends a confirmation email via SendGrid. No scheduled needed — both triggers are event-driven.

⚡ /backend.js — Stripe webhook + email notification
export async function fetch(request, ctx) { const url = new URL(request.url) if (url.pathname !== "/webhook/stripe") return new Response("Not Found", { status: 404 }) const sig = request.headers.get("stripe-signature") const body = await request.text() const event = verifyStripeWebhook(body, sig, ctx.secrets.STRIPE_WEBHOOK_SECRET) if (event.type === "checkout.session.completed") { await ctx.db.put({ _id: `order-${event.data.object.id}`, type: "order", customerEmail: event.data.object.customer_email, amount: event.data.object.amount_total, status: "paid", stripeSessionId: event.data.object.id }) } return new Response("OK") } export async function onChange(event, ctx) { if (event.doc?.type !== "order" || event.oldDoc) return await fetch("https://api.sendgrid.com/v3/mail/send", { method: "POST", headers: { Authorization: `Bearer ${ctx.secrets.SENDGRID_API_KEY}`, "Content-Type": "application/json" }, body: JSON.stringify({ personalizations: [{ to: [{ email: event.doc.customerEmail }] }], from: { email: "orders@example.com" }, subject: `Order Confirmation #${event.doc._id}`, content: [{ type: "text/plain", value: `Thank you! Amount: $${event.doc.amount / 100}` }] }) }) }

Runtime Architecture — Per-App BackendDO

Backend handlers execute server-side inside a Durable Object. Each vibe with a /backend.js file gets a BackendDO keyed by {ownerHandle}/{appSlug}. The DO loads the handler source, compiles it with new Function() (via the unsafe_eval compatibility flag already present for AccessFnDO), and dispatches to the matching export on each trigger.

Why not QuickJS. The access function uses QuickJS WASM for maximum sandbox isolation — but access functions are synchronous, short-lived, and make no outbound calls. Backend handlers are the opposite: they need real fetch(), async/await, and timer integration. Bridging async operations through QuickJS host functions is complex and fragile. The per-DO model provides sufficient isolation — each vibe runs in its own DO with its own memory, and Cloudflare's Worker runtime provides the security boundary.

⚡ BackendDO — dispatch architecture
class BackendDO extends DurableObject { handlers = null // compiled { fetch?, scheduled?, onChange?, config? } alarmInterval = null // parsed from config.scheduled.interval async loadSource(source) { // compile once, cache in memory — eviction reloads from D1 const module = new Function("exports", "fetch", source + "; return exports;") this.handlers = module({}, globalThis.fetch) } // HTTP trigger — routed from Worker fetch handler async fetch(request) { if (this.handlers?.fetch) { const ctx = this.buildCtx(request) return this.handlers.fetch(request, ctx) } return new Response("No fetch handler", { status: 404 }) } // Timer trigger — Durable Object alarm async alarm() { if (this.handlers?.scheduled) { const ctx = this.buildCtx(null) await this.handlers.scheduled({ scheduledTime: Date.now() }, ctx) } // re-arm alarm for next tick if (this.alarmInterval) { await this.ctx.storage.setAlarm(Date.now() + this.alarmInterval) } } // Database trigger — called from write path after doc stored async invokeOnChange(event) { if (this.handlers?.onChange) { const ctx = this.buildCtx(null) await this.handlers.onChange(event, ctx) } } }

Source loading. At push time, processBackendBindings() detects /backend.js in the vibe filesystem (same pattern as /access.js detection) and stores a BackendFunctionBindings row with the asset CID. On first request to the DO, it loads the source from the asset store and compiles it. The compiled handlers are cached in DO memory — Cloudflare evicts the DO when idle, and the next request reloads and recompiles.

Alarm lifecycle. When a vibe with a scheduled export is pushed, the binding creation also arms the first alarm. The alarm fires, the handler runs, and the DO re-arms for the next interval. If the vibe is updated with a new interval, the next alarm uses the new value. If scheduled is removed, no alarm is re-armed and polling stops. Deleting the vibe destroys the DO and all its alarms.

Isolation model. Each vibe's BackendDO is its own Durable Object — separate memory, separate storage, separate alarm schedule. Cloudflare's runtime provides the security boundary between vibes. The unsafe_eval compatibility flag is required for new Function() compilation, but the evaluated code runs within the Worker's security context, not with elevated privileges. The DO cannot access other vibes' data, other DOs, or Worker-level environment bindings beyond what ctx provides.

ctx.db writes go through access.js. When ctx.db.put() is called from a backend handler, the write follows the same path as a client write — including access function evaluation if /access.js exists. The write identity comes from the trigger: onChange passes through the original writer's userHandle, fetch uses the session-authenticated user, and scheduled defaults to the vibe owner. A handler can override with ctx.db.put(doc, { as: "handle" }) — only the vibe owner's code can impersonate. The access function always sees a real user.userHandle, never a special system identity. No backdoor, no new API surface on access.js.

LLM codegen docs (backend-js.llms.md). The codegen prompt is the primary delivery mechanism for the backend.js API. LLMs learn APIs from examples, not type signatures — a well-chosen code sample teaches the model more than a page of documentation. The backend-js.llms.md file should be built around complete, copy-adaptable examples for each handler pattern: a fetch handler with a simple path router, a webhook receiver with HMAC signature verification, an OAuth code exchange, a scheduled poller writing to ctx.db, an onChange handler that filters on doc type and calls an external API. Each example should be minimal but fully working — no pseudocode, no "implement here" placeholders. The LLM copies the shape and adapts the details to the user's prompt.

Every vibe knows its own ownerHandle and appSlug; the webhook URL follows mechanically from those. When the generated app needs to register a webhook URL with a third-party service (Stripe, Sonos, GitHub) or set an OAuth redirect URI, the LLM should generate UI or setup instructions that show the user their webhook endpoint.

Summary. /backend.js gives vibes server-side capabilities through three named exports: fetch for HTTP triggers (webhooks, OAuth, APIs), scheduled for timer-driven polling (with intervals declared in config), and onChange for database-reactive side effects (notifications, external sync). All handlers share a ctx object with database access (ctx.db), app-level secrets (ctx.secrets), and user identity (ctx.userInfo). Writes go through /access.js as the trigger's identity — onChange passes through the original writer, fetch uses the session user, scheduled defaults to the vibe owner. Handlers can override with ctx.db.put(doc, { as: "handle" }). The access function always sees a real userHandle, never a system identity. Execution happens in a per-vibe Durable Object (BackendDO) using new Function(). onChange events are delivered via Worker Queue for reliable retry. The file is discovered automatically at push time, following the same pattern as /access.js detection.
