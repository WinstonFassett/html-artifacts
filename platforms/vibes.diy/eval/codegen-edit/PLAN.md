# `eval/codegen-edit` — generation-stream archive & analysis harness

## Goal

Build a repeatable harness that runs the CLI's generation flow N times against
curated prompts, archives every section event so runs are **fully replayable**
through the parser offline, and surfaces the runs that produced edit-apply
errors as a corpus for parser/prompt hardening.

The CLI generate path and the chat web view share the same parse/apply
primitives (`parseFenceBody`, `applyEdits`, `applyReplace` from
`@vibes.diy/call-ai-v2`), so an apply error reproduced via the CLI is the same
error a chat-view user would hit. The CLI is the cheaper place to drive
high-volume eval against it.

---

## Decisions (locked in before coding)

- **Auth**: reuse existing CLI login. Pin runs to the existing `eval` user slug
  via the CLI's already-supported `--handle` flag
  ([generate-cmd.ts:240-246](../../vibes-diy/cli/cmds/generate-cmd.ts#L240-L246)).
  No new credentials, no new account. Generated apps land under the `eval`
  namespace where they don't pollute the real user history.
- **API endpoint**: prod. The corpus needs to reflect the failure surface real
  users hit today.
- **In-process, not subprocess.** Import `vibesDiyApiFactory`,
  `chat.sectionStream`, and `resolveSectionStream` directly from the CLI's
  building blocks rather than spawning `vibes-diy generate`. Reasons:
  - Tee the section stream into both an archive writer and the resolver in one
    pass — no re-parse, no risk of drift.
  - Capture every `SectionEvent` and every `ResError` envelope, not just what
    the CLI's stderr happened to print.
  - Avoid building eval-only flags onto the user-facing CLI.
- **Sequential runs.** Parallelizing risks rate-limit errors that pollute the
  apply-error corpus with infrastructure noise.
- **Corpus theme: callAI + Fireproof, edit-heavy.** The interesting failures
  are in multi-block SEARCH/REPLACE traffic, which means apps that get edited
  across multiple turns. Avoid ImgVibes/image-gen prompts — they bias toward
  single-create flows and aren't where the parser hurts. The seed prompt is a
  task tracker with AI auto-tagging and subtask generation; we vary the
  surface (calendar, kanban, notes, journaling, recipes, …) but keep the
  callAI + Fireproof core constant.

## Workspace layout

```
eval/codegen-edit/
  package.json              # private workspace, add to root pnpm-workspace.yaml
  PLAN.md                   # this file
  README.md
  src/
    run.ts                  # one-shot: take prompt, archive a single run
    batch.ts                # driver: loop over prompts, write to archive root
    replay.ts               # offline: re-run parser on archived sections
    analyze.ts              # aggregate stats / dump failures
    archive.ts              # archive read/write primitives + types
    prompts.ts              # curated prompt list + prompt source loader
  prompts/
    seed.jsonl              # initial 20 prompts (one per line)
  archive/                  # gitignored — generated runs
    .gitignore
  findings/                 # markdown reports from analyze.ts
```

## Per-run archive layout

```
archive/<timestamp>_<slug>/
  manifest.json       # prompt, userHandle, appSlug, apiUrl, model (if surfaced),
                      #   streamId, startedAt, finishedAt, exitState, turnCount
  sections.jsonl      # every SectionEvent + ResError, one per line, in order
  prompt-events.jsonl # promptId, promptBlockEnd timing per turn
  resolved/           # final files written by resolveSectionStream
    <path>...
  errors.json         # FsApplyErrorMsg list with full search/replace bodies
                      #   + summarized failures
  upstream-errors.json
  notes.md            # optional, hand-written
```

`sections.jsonl` is the **load-bearing artifact** — single source of truth for
replay. Everything else (resolved files, errors) is derived and can be
regenerated by `replay.ts`.

## Replay-first principle

`analyze.ts` and any future parser-tweak iteration must read `sections.jsonl`
and re-run `createFileSystemStream` (or `parseFenceBody`/`applyEdits`
directly) — never trust `errors.json` as ground truth. This means: tweak the
parser → run replay over the archive → diff old vs new error counts. No
regeneration cost to iterate on the parser.

## What we explicitly DON'T archive

- The deployed bundle. Skip `pushFromDir` entirely from the harness.
- READMEs.
- Anything from the iframe / sandbox runtime side. Scope is model→parser.

## Prompt corpus design

**Theme**: every prompt asks for a callAI + Fireproof app that requires
multi-turn edits. The seed prompt is the user's reference example (task
tracker with AI auto-tagging and subtask generation). Variety comes from
swapping the surface domain and the AI augmentation, not from changing the
stack.

**Multi-pass structure.** A single one-shot generation can't produce
SEARCH/REPLACE traffic — the model has nothing to replace against. So each
prompt entry expands to a **turn sequence**: an initial create, then 2-3
follow-up edit prompts in the same chat. This is what produces the heavy
SEARCH/REPLACE volume where the parser actually fails. (See "Limitation: no
edit command yet" below for how we deliver the follow-up turns.)

**Seed prompts (20)** — domains × AI augmentations, all callAI + Fireproof:

- task tracker w/ AI auto-tagging + subtask generation (canonical)
- kanban board w/ AI priority scoring + duplicate detection
- recipe book w/ AI ingredient substitutions + meal planning
- journaling app w/ AI sentiment tagging + weekly summaries
- bookmark manager w/ AI topic clustering + read-time estimates
- meeting notes w/ AI action-item extraction + follow-up reminders
- reading list w/ AI summarization + related-book suggestions
- workout log w/ AI progression suggestions + form-cue retrieval
- expense tracker w/ AI categorization + budget anomaly detection
- gardening journal w/ AI plant-care tips + seasonal task generation
- language flashcards w/ AI example-sentence generation + difficulty grading
- contact CRM w/ AI relationship-strength scoring + follow-up suggestions
- pantry inventory w/ AI recipe suggestions from on-hand items
- habit tracker w/ AI streak coaching + habit-stacking suggestions
- dream journal w/ AI symbol interpretation + recurring-theme detection
- travel itinerary w/ AI activity suggestions + packing list generation
- code-snippet vault w/ AI tag suggestions + similar-snippet retrieval
- music practice log w/ AI exercise suggestions + progression tracking
- gift-idea tracker w/ AI suggestion generation per recipient profile
- daily standup log w/ AI blocker-detection + weekly summary

Each entry in `prompts/seed.jsonl` is:

```json
{ "id": "task-tracker", "create": "...", "edits": ["...", "...", "..."] }
```

Editing prompts deliberately ask for things that touch existing code:
"add a dark mode toggle", "show subtask completion as a progress bar",
"persist the AI-suggested tags in Fireproof and render a tag cloud."

## Failure heuristic

A run is "interesting" if any of:

- `errors.json` non-empty (parse or apply errors).
- `upstream-errors.json` non-empty.
- Resolved file count is zero.
- Resolved files exist but don't include `export default` (silent malformed).

## Limitation: no `edit` command in the CLI yet

The CLI today only has `generate`, which opens a fresh chat. To deliver
follow-up edit turns we need an `edit` command (or a `--continue` mode on
generate) that opens an existing chat by `appSlug` and sends another user
prompt.

Filed as a separate issue (see VibesDIY/vibes.diy issue link in the PR that
introduces this harness). **Until that lands**, v1 of the harness can:

- (a) stub edit turns by calling `api.openChat({ userHandle, appSlug, mode: "chat" })` directly with the existing slug — the SDK supports this, the CLI just doesn't expose it. This is fine for the harness since we're already in-process.
- (b) generate single-turn-only and accept the lower error rate.

Recommend (a) — it gives us the multi-pass corpus immediately without waiting
on a CLI feature, and validates the SDK shape the new `edit` command will
ultimately use.

## Implementation steps

### Step 1 — Wire the workspace

- Add `eval/codegen-edit` to root `pnpm-workspace.yaml`.
- `package.json`: `private: true`, `name: @vibes.diy/eval-codegen-edit`,
  depend on `@vibes.diy/api-types`, `@vibes.diy/call-ai-v2`,
  `@vibes.diy/api-pkg`, `@adviser/cement`. Reuse the same
  `vibesDiyApiFactory` the CLI uses.
- Scripts: `pnpm -F @vibes.diy/eval-codegen-edit run gen`, `replay`, `analyze`.
- `archive/.gitignore` to keep raw runs out of git (prompt + model output may
  be sensitive).

### Step 2 — Implement `run.ts` (single-run archiver)

1. Read auth from existing CLI credentials store.
2. `api.openChat({ userHandle: "eval", prompt, mode: "chat" })`, then
   `chat.prompt(...)`.
3. Tee `chat.sectionStream`:
   - branch A → `sections.jsonl` writer (JSON.stringify each event, `\n`).
   - branch B → `resolveSectionStream` with `onSnapshot`/`onError` writing
     into `errors.json`.
4. For multi-turn entries: after first `prompt.block.end`, send the next
   user prompt via the same chat; each turn appends to the same
   `sections.jsonl` with a turn boundary marker.
5. On final `prompt.block.end`, close streams, `chat.close()`, write
   `manifest.json`.
6. Exit code: 0 always — apply failures are data, not errors. Auth/network
   failures are surfaced via `manifest.json.exitState`.

### Step 3 — Implement `batch.ts`

- Read `prompts/seed.jsonl`.
- For each entry: call `run.ts` logic, append to top-level
  `archive/index.jsonl`.
- Print one-line summary: `[ok|errors=N|upstream=N|empty] slug — first N chars`.
- `--filter errored` mode: re-run only prompts whose previous run had errors.

### Step 4 — Implement `replay.ts`

- Input: archive directory.
- Read `sections.jsonl`, feed through `createFileSystemStream`, write derived
  outputs to `replay-<timestamp>/` next to original.
- Diff against original `resolved/` and `errors.json` — should be byte-
  identical for v1; divergence means parser changed.
- This is the iteration loop for parser hardening.

### Step 5 — Implement `analyze.ts`

First-pass analyses (cheap, all from `sections.jsonl` + `errors.json`):

- Counts: total / with-errors / empty-resolution / upstream-error.
- Histogram of failure `kind` (parse vs apply, by `reason`).
- Failure clusters: group by normalized `search` prefix to find recurring
  patterns.
- Per-run "minimal repro": the failing section's raw `code.line` events
  around the failed fence — what we hand to a parser test or model-prompt
  tweak.

Output a markdown report listing top failure modes with one example archive
path each, written to `findings/<date>.md`.

### Step 6 — Generate the corpus

- Run `batch.ts` against the 20-prompt seed list. Sequential, ~3-7 min/run
  for 4-turn entries, expect 1-2 hours total.
- If error rate is too low, expand prompts toward more aggressive
  edits-on-existing-code in a follow-up batch.
- Stop when we have ~5+ runs with apply errors covering distinct failure
  modes.

### Step 7 — Analyze and feed back

- Run `analyze.ts`, write findings to `findings/<date>.md`.
- For each distinct failure mode: decide whether the fix is in the parser
  (`@vibes.diy/call-ai-v2` `parseFenceBody`/`applyEdits`) or in the prompt
  template (system prompt that teaches SEARCH/REPLACE format).
- Lift minimal repros into unit tests in `@vibes.diy/call-ai-v2` so future
  parser changes can't regress on this corpus.

## Out of scope (for v1)

- Compilation/runtime correctness of the generated apps. We're scoring the
  parser, not the model's React skills.
- Cross-model comparison. Current model only.
- Auto-replay on parser change in CI. Manual for now; if the corpus proves
  useful, wire as a CI job later.
